Inventory Your list of managed machines
What is inventory?
1

It's your address book

Ansible needs to know which machines to manage. The inventory file is that list — hostnames or IPs, optionally grouped.

2

Groups let you target subsets

Instead of running tasks on every host, you target a group: hosts: webservers runs only on machines in that group.

3

Two built-in groups always exist

all — every host in inventory. ungrouped — hosts not in any explicit group.

4

Attach variables to hosts or groups

Connection details (user, port, SSH key), environment flags, app config — all can live in inventory as variables.

Mental model
Static INI — most common format
# hosts.ini

# ── ungrouped host ──────────────────
bastion.example.com

# ── group: webservers ───────────────
[webservers]
web-01 ansible_host=10.0.1.1
web-02 ansible_host=10.0.1.2

# ── group: databases ────────────────
[databases]
db-01 ansible_host=10.0.2.1 ansible_port=2222
db-02 ansible_host=10.0.2.2

# ── group of groups ─────────────────
[production:children]
webservers
databases

# ── group-level variables ───────────
[webservers:vars]
ansible_user=ubuntu
http_port=80

# ── global variables ────────────────
[all:vars]
ansible_python_interpreter=/usr/bin/python3
Line-by-line breakdown
1

[webservers] — a group header

Square brackets define a group name. Every host listed below (until the next header) belongs to it.

2

ansible_host= — connection address

The alias on the left (web-01) is the display name. ansible_host is the real IP Ansible connects to.

3

[production:children] — group of groups

Lets you target all webservers AND databases at once with hosts: production. Hosts are inherited.

4

[webservers:vars] — group variables

Sets variables for every host in that group. Overrides [all:vars] for those hosts.

Inline vars like ansible_port=2222 are fine for connection details. For application vars, prefer group_vars/ files (see Host & group vars tab).
YAML inventory — same info, cleaner syntax
# hosts.yml
all:
  vars:
    ansible_python_interpreter: /usr/bin/python3

  children:

    webservers:
      vars:
        ansible_user: ubuntu
        http_port: 80
      hosts:
        web-01:
          ansible_host: 10.0.1.1
        web-02:
          ansible_host: 10.0.1.2

    databases:
      hosts:
        db-01:
          ansible_host: 10.0.2.1
          ansible_port: 2222
        db-02:
          ansible_host: 10.0.2.2

    production:
      children:
        webservers:
        databases:
YAML structure map
A

all: — root key, always required

Everything nests under all. Vars here apply to every host.

B

children: — define groups

Each key under children is a group name. Groups can contain hosts, vars, and more children.

C

hosts: — machines in this group

Each key is a hostname alias. Host-specific vars go as nested keys under it.

D

Group of groups in YAML

Use children: inside a group and list other group names. Leave their value empty (webservers:) — Ansible fills it in.

YAML and INI are equivalent — use whichever you prefer. Many teams prefer YAML for larger inventories because nesting makes relationships explicit.
Where to put variables (best practice)
# Recommended folder layout next to inventory:

inventory/
├── hosts.ini
├── group_vars/
│   ├── all.yml         ← applies to every host
│   ├── webservers.yml  ← only webservers group
│   └── databases.yml   ← only databases group
└── host_vars/
    ├── web-01.yml      ← only this one host
    └── db-01.yml       ← only this one host
# group_vars/webservers.yml
http_port: 80
max_connections: 1000
nginx_version: "1.24"

# host_vars/web-01.yml
backup_primary: true   ← only this host is primary
1

Variable precedence (low → high)

group_vars/all → group_vars/groupname → host_vars/hostname → inline host vars → extra vars (-e). More specific = higher priority.

2

Never hardcode secrets in inventory

Use ansible-vault encrypt on your vars files, or reference environment variables. Commit encrypted files, not plain text.

3

Common host connection vars

ansible_host ansible_user ansible_port ansible_ssh_private_key_file ansible_become ansible_python_interpreter
Dynamic inventory — hosts from an API
# Use a plugin or script instead of a file

# aws_ec2.yml — auto-discover EC2 instances
plugin: amazon.aws.aws_ec2
regions:
  - us-east-1
filters:
  instance-state-name: running
keyed_groups:
  - key: tags.Role      ← group by EC2 tag
    prefix: role
  - key: placement.region
    prefix: aws_region

# run with:
# ansible-inventory -i aws_ec2.yml --list
# ansible-playbook -i aws_ec2.yml play.yml
When to use dynamic inventory
1

Cloud / autoscaling environments

When instances spin up and down, a static file gets stale fast. A dynamic plugin queries the cloud API each run.

2

Popular plugins

amazon.aws.aws_ec2google.cloud.gcp_computeazure.azcollection.azure_rmcommunity.vmware.vmware_vm_inventory
3

Inspect what it finds

Run ansible-inventory -i aws_ec2.yml --graph to see the group/host tree before running a playbook.

Use a role in a playbook
# playbook.yml
- name: Configure web servers
  hosts: webservers
  roles:
    - nginx              ← simple include
    - role: nginx         ← with overrides
      vars:
        nginx_port: 8080
Details

roles/ — the roles directory

Place this folder next to your playbook. Ansible automatically looks here for roles by name. Each subfolder is one role.

# Convention: one role = one concern
# nginx  →  installs & configures nginx
# mysql  →  installs & configures MySQL
# users  →  manages system users
Folder responsibilities at a glance
T

tasks/

The main job. Ansible runs tasks/main.yml automatically when the role is included. Split large task lists into sub-files and import_tasks them from main.

H

handlers/

Triggered by notify: in tasks. handlers/main.yml is loaded automatically. Always use the full handler name when notifying.

J

templates/

Jinja2 (.j2) files. Use the template module to render and push to hosts. Variables from vars/defaults are available inside.

D

defaults/ vs vars/

defaults/main.yml — lowest priority, safe to override. vars/main.yml — higher priority, harder to override. Put user-tunable settings in defaults.
