ReplicaSet manifest — annotated
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: my-app-rs
  labels:
    app: my-app  # [1] tag ON the RS object itself
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app  # [2] RS's search filter — watches pods with this label
  template:
    metadata:
      labels:
        app: my-app  # [3] label stamped on every pod created
    spec:
      containers:
        - name: my-app
          image: nginx:1.25
[1] metadata.labels
Tag on the RS object
Optional. Lets other objects (like a Service) find this RS. Does not affect pod creation.
[2] selector.matchLabels
RS's search filter
RS counts pods with this label. Too few → create. Too many → delete. Must match [3].
[3] template.labels
Stamp on every new pod
Baked into each pod at birth. Must match [2] — Kubernetes rejects manifest if they differ.
Key rules to remember
⚠
[2] and [3] must always match. If they differ, Kubernetes rejects the manifest immediately.
⚠
Standalone pods with matching labels get adopted. RS counts them as its own replicas — can cause unexpected behavior.
i
In practice, use Deployments, not bare ReplicaSets. A Deployment wraps an RS and adds rolling updates + rollback.
i
[1] metadata.labels is optional. The RS works fine without it — only needed when other objects need to target the RS.
Useful commands
# apply manifest
kubectl apply -f replicaset.yml
# check RS and pods
kubectl get rs
kubectl get pods
# scale up or down
kubectl scale rs my-app-rs --replicas=5
# inspect events
kubectl describe rs my-app-rs
# delete RS and its pods
kubectl delete rs my-app-rs
